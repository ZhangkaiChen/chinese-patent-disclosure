# 项目扫描规范

## 扫描目标

项目扫描的目的不是罗列所有文件，而是快速建立“论文/代码/文档中的真实技术方案”。扫描结果应能回答：要解决什么技术问题，采用哪些模块，数据如何流动，训练和推理如何执行，输出什么结果，哪些内容可能形成专利点。

## 读取优先级

优先读取以下材料：

1. 用户明确指定的 DOCX、PPTX、PDF、论文、代码文件或目录；
2. 专利相关文档，例如专利点分析、已有交底书、申报建议、创新点汇总；
3. 详细设计、技术方案、流程图、技术对比分析；
4. 核心实现代码，例如算法、模型、数据处理、规则、接口、训练和推理入口；
5. 系统设计、架构说明、模块划分、数据流与控制流；
6. 测试脚本、部署脚本、可视化脚本和业务说明。

不要在未扫描核心材料前直接撰写。若材料很多，先读取最能确定技术路线的文件，再按问题补充读取。

## DOCX / PPTX 处理

DOCX 和 PPTX 不应只凭文件名判断内容。应先转为 Markdown、纯文本或结构化提取结果，再扫描。若 skill 或项目中存在转换脚本，可使用对应工具；否则使用可用文档处理能力抽取文本。

处理要求：

- DOCX：提取标题、正文、表格、公式附近文字、图题和批注/修订痕迹；
- PPTX：提取每页标题、正文、图示文字、备注、表格和流程图文本；
- 旧版 `.doc`、`.ppt` 不能直接按 OOXML 处理，应提示需要另存为 `.docx`、`.pptx` 或改用其他可读取格式；
- 若存在公式图片或 MathType/Word 公式，应记录公式所在章节及其变量解释是否完整；
- 若文档是模板，应提取模板格式，不机械保留旧技术内容。

## 代码扫描重点

从代码中重点确认：

- 模型输入、输出、张量维度和任务类型；
- 网络模块名称、连接关系、注意力/门控/掩码/对齐/蒸馏等机制；
- 数据预处理、样本构建、标签组织和增强策略；
- 训练损失、权重参数、阈值、推理路径和异常处理；
- 与论文描述不一致的实现细节。

代码是核实技术方案的证据，不要把未实现、未说明的模块写成既有方案。

## 图片与独立素材

独立 `.png`、`.jpg`、`.webp` 等图片目录通常不作为项目扫描阶段逐个 OCR 的对象。若用户点名图片路径，或图片未出现在已转换 Office 文档中且明显承载关键技术流程，再单独查看。

Office 文档转换后，嵌入图通常已在 Markdown 图片引用中体现；扫描应以转换后的文字、表格、图题和必要图片为依据，避免重复读图造成噪声。

## 扫描输出

扫描后应形成简洁工作摘要，通常包括：

- 技术主题和应用场景；
- 输入数据和输出结果；
- 核心模块和数据流；
- 训练、推理和部署流程；
- 候选专利点；
- 需要进一步查证的问题。

该摘要可作为内部工作材料，不一定写入专利正文。
