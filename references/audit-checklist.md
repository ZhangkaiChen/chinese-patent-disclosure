# 专利审查清单

## 范围一致性

检查最新稿件或待撰写材料是否满足：

- 专利标题与技术方案、下游输出一致；
- 背景、技术方案、公式、附图、关键点和保护点中的模态范围一致；
- 下游任务在技术领域、技术方案、损失函数、推理函数和附图中一致；
- 历史版本中已删除的模块、任务或模态没有被重新加入；
- “可替代方案”没有把保护范围扩展到主方案没有支撑的内容；
- 若用户提供模板，最终文档格式与模板一致，内容没有残留模板旧技术。

## 技术逻辑

审查整体架构是否形成清晰流程。常见顺序为：

1. 多源输入及质量/有效性信息；
2. 空间或时间组织与对齐；
3. 模态专属编码；
4. 跨分辨率统一 Token 表示；
5. 缺模态鲁棒训练；
6. 掩码交叉注意力与质量门控融合；
7. Token 对齐与视图一致性约束；
8. 任意可用模态下的动态推理；
9. 下游任务头输出；
10. 置信度或质量评价输出。

上述顺序仅为审查参考。若最新材料中没有某一步，不要自动加入，应指出缺口或按最新范围处理。

## 公式审查

逐个公式检查：

- 公式编号是否被附近正文引用；
- 每个变量、下标、集合是否解释；
- 样本索引、模态索引、任务索引、Token 索引、网格索引、时间索引是否区分清楚；
- 掩码、质量评分、概率、阈值、损失权重、温度系数等是否给出取值范围；
- 必要时是否说明张量形状、序列长度或广播方式；
- 可选损失项是否明确为可选，而不是所有实施例必需；
- 函数是可学习函数、规则函数还是二者均可；
- 公式没有引入正文未出现的模块或任务；
- 正文没有描述公式未覆盖的关键机制。

常见问题包括：

- 同一符号同时表示质量评分和 Query 矩阵；
- 定义了任务集合，后文又使用不同任务下标；
- 最新稿已删除某模态或任务，但公式说明中仍出现该模态或任务；
- 声称 Softmax 前屏蔽无效连接，但公式实际在归一化后才处理；
- 缺少阈值、截断上下限、温度系数、损失权重解释；
- 写了完整视图教师信号，但没有说明停止梯度、教师来源或一致性距离函数。

## 语言审查

将空泛表述改为具体机制：

- “提高智能化水平”改为具体模块和输出；
- “实现高效融合”改为注意力、门控、池化或对齐机制；
- “增强鲁棒性”改为处理哪类缺失、噪声、低质量或跨域情况；
- “多任务泛化能力强”改为哪些任务头如何共享统一表示。

推荐使用：

- “本申请通过……使……”
- “该结构能够……”
- “从而避免……”
- “在不改变……的情况下……”

## 关键点和保护点审查

建立两列映射：

- 左列：关键点标题及一句话技术核心；
- 右列：欲保护点标题及拟保护机制。

需要标记的问题：

- 关键点是宽泛目标，保护点却是具体模块，二者不对应；
- 保护点引入了关键点中没有的技术特征；
- 两个保护点内容高度重叠；
- 关键点只是数据收集或数据集构建，没有形成技术机制；
- 最有价值的机制被埋在过长的框架性条目中；
- 保护点与现有论文或现有专利的主贡献重合，缺少进一步限定。

## 现有技术区分度审查

每个技术优点都应包括：

- 对比的现有技术或技术类别；
- 该技术的假设、适用范围或不足；
- 本申请采用的具体技术机制；
- 因结构或流程差异产生的技术效果。

避免只写“本申请效果更好”。必须把效果绑定到具体结构差异。

## 审查输出格式

只审查不修改时，按以下顺序输出：

1. 影响专利性或技术正确性的主要问题；
2. 公式与变量问题；
3. 技术范围和术语一致性问题；
4. 重复、冗余和 AI 味表述问题；
5. 必要的替换句或修改建议；
6. 整体技术架构总结。

意见要具体，尽量引用章节名、公式编号或原文关键词。
