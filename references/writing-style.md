# 撰写风格规范

## 标准交底书结构

除非用户提供了其他公司模板，否则优先采用以下结构：

1. 发明名称
2. 一、详细介绍技术背景，并描述已有的与本发明最相近的实现方案
3. 二、现有技术的缺点是什么？
4. 三、本发明技术方案的详细阐述，应该结合附图进行说明
5. 四、本发明的关键点和保护点是什么？
6. 五、上述技术方案是否有替代方案？
7. 六、与最接近的现有技术相比的技术优点
8. 七、其他有助于理解本申请的技术资料

如果用户提供现有专利或交底书模板，应严格使用该模板的章节结构、标题层级、段落顺序和常用表述，不强行套用上述通用结构。

## 只提供论文或代码时的撰写方式

没有现有专利模板时，也应能够直接撰写专利。处理顺序为：

1. 从论文摘要、方法、模型结构、损失函数、实验动机和消融实验中提取技术问题和技术手段。
2. 从代码中核对真实模块名称、数据流、输入输出、训练策略、推理逻辑和可选参数。
3. 将论文或代码中的“方法贡献”转化为专利中的“技术问题 - 技术方案 - 技术效果”。
4. 对论文中已经属于公开常规做法的内容，只作为背景或实施方式，不作为核心创新点。
5. 对代码中体现的工程闭环、模块组合关系、动态推理、质量控制、损失约束、参数选择机制等，优先判断是否可形成保护点。

## 技术模块写法

每个技术模块按以下顺序展开：

1. 问题或场景：例如“针对……问题”。
2. 技术手段：例如“设计/采用/构建……方法、机制或模块”。
3. 实施细节：例如“通过……，并在……阶段……”。
4. 输出结果：例如“得到……/输出……/生成……”。
5. 技术效果：例如“从而……，解决……问题”。

一个合格段落应能回答：输入是什么、执行了什么处理、输出是什么、供哪个后续模块使用、解决了哪个现有技术缺陷。

## 关键点写法

采用“短标题 + 扩展描述”的形式。常用句式为：

“本申请针对……问题，设计……方法/机制/框架，通过……，实现……，解决……问题。”

不要把单纯数据集构建、数据收集或常规预处理直接作为核心创新点，除非其与训练、融合、推理、质量控制或任务输出形成不可分割的技术机制。

示例：

“缺模态感知的掩码交叉注意力与质量门控融合机制
本申请针对普通交叉注意力机制默认所有Token均有效、缺失占位值或低质量Token可能污染注意力归一化等问题，设计缺模态感知的交叉注意力融合机制。在计算注意力权重时，将模态有效性和Token有效性转换为掩码偏置，并在Softmax之前屏蔽无效连接，使不存在观测基础的Token不参与注意力归一化。同时，引入质量门控权重，根据模态质量、有效覆盖比例和任务相关性动态调节各模态贡献，实现有效模态增强与低质量模态抑制。”

## 欲保护点写法

欲保护点应与关键点对应，但可以比关键点更偏向权利要求语言。每一项应包括：

- 要保护的技术特征，而不是泛泛的效果；
- 针对的技术缺陷；
- 必要组成部分和相互作用关系；
- 产生的技术结果；
- 可替代实施方式的边界。

避免把已有论文或现有专利已经公开的“多模态统一表征”“跨模态对齐”“注意力融合”等宽泛概念直接写成本申请独占创新。应把保护重心放在具体机制组合和部署闭环上。

## 常用专利语气

推荐使用：

- “本申请涉及……技术领域”
- “本申请适用于……场景”
- “现有技术通常采用……”
- “然而，在实际应用中……”
- “本申请要解决的技术问题是……”
- “在训练阶段/融合阶段/推理阶段……”
- “与……相比，本申请……”

避免使用：

- “AI大模型赋能”“智能化”“革命性”“颠覆式”
- 没有机制支撑的“极大提升”“完美解决”“显著增强”
- 没有解释的“深度融合”“高效协同”“端到端智能识别”
- 与最新材料范围不一致的任务清单或模态清单

## 公式上下文规则

公式前应先说明该公式解决什么问题、用于哪个模块。公式后必须解释：

- 每个变量、下标和集合；
- 取值范围、张量维度或序列长度；
- 函数是规则函数、可学习函数还是二者均可；
- 该公式的输出如何进入下一步骤；
- 可选项是否为必要特征。

若公式为损失函数，应解释正样本、负样本、教师信号、学生输出、任务权重、可选损失项及适用任务。
