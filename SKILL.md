# 论文代码转专利

## 用途

使用本 skill 将论文、源代码、模型结构、实验材料、算法说明、技术方案、DOCX/PPTX 草稿或现有专利模板整理为中文发明专利交底书。输出应符合实用型专利交底书风格：技术问题明确，技术手段具体，模型结构和公式上下文完整，创新点具有与现有技术的区分度，便于代理人进一步拆分权利要求。

## 基本原则

不要凭记忆撰写。每次开始前必须先读取用户本次提供的最新 DOCX、PPTX、PDF、代码、论文或技术材料。若历史对话与最新材料不一致，以最新材料为准。

现有专利不是必需输入。若用户只提供论文、代码或算法说明，也应从材料中提炼技术问题、技术方案、网络模块、训练/推理流程、公式、实验动机和可保护创新点，形成完整交底书。若用户提供现有专利、交底书或模板，则先读取模板结构和行文风格，并严格按照该模板的章节格式、标题层级和表述习惯进行总结、改写或扩写。

保持流程轻量化。正式成稿或涉及现有技术章节时执行完整流程；快速润色、局部公式审查、附图修改或小范围修订时，只加载必要参考文件，不强制执行查新和全量自检。

用户只要求审查时，不直接修改文件；应输出具体问题和专业修改建议。用户明确授权修改时，才能改写 DOCX 或其他交付文件，并按要求高亮、修订模式或另存版本。

## 轻量六步流程

1. **项目扫描**
   - 按 `references/project_scan.md` 读取材料：先读 README、论文、技术说明、现有草稿和核心代码；DOCX/PPTX 先转 Markdown 或抽取文本后扫描。
   - 提取真实技术范围，包括模态、任务、网络结构、公式、参数、数据流、训练推理流程和输出结果。

2. **候选专利点讨论与融合**
   - 按 `references/patent_point_fusion.md` 从论文或代码中提炼候选点，剔除常规做法和现有技术主贡献，合并重复点。
   - 只有具备明确技术问题、具体机制、材料支撑和现有技术区分度的内容，才进入关键点和保护点。

3. **查新与现有技术定位**
   - 正式成稿或撰写现有技术章节时，按 `references/prior_art_search.md` 执行轻量查新。
   - 优先使用国知局中国专利公布公告检索；若本 skill 或项目中存在可用的 `cnipa_epub_search.py`，优先使用该脚本。异常、无结果或需要论文对比时，降级使用 WebSearch、Google Scholar 或 Google Patents。
   - 查新记录写入工作材料或用于第一章现有技术，不把完整检索过程写入正文；不得虚构专利公开号或论文结论。

4. **交底书成稿**
   - 按 `references/writing-style.md` 撰写，现有用户模板优先于通用模板。
   - 每个技术模块按“输入 - 处理 - 输出 - 作用/效果”展开，公式满足 Word 公式或 MathType 可转换要求，并解释所有变量。

5. **自检**
   - 按 `references/disclosure_self_check.md` 检查逻辑闭环、公式与参数一致、术语范围一致、关键点/保护点对应和 AI 味表述。
   - 自检结果只作为修订依据，不写入专利正文。

6. **迭代修改**
   - 按 `references/iteration_policy.md` 执行：小改直接在 Word 中高亮或修订；大版本修改、方向纠正或结构重写另存新文件。

## 输出要求

- **计划类请求**：先给简明撰写或修改计划，等待用户确认后再执行。
- **撰写类请求**：输出完整交底书，或在用户授权时直接更新 DOCX。
- **审查类请求**：先列具体问题，再给对应修改建议，必要时附可替换表述。
- **公式审查**：逐条列出问题公式、未解释变量、符号冲突、公式与文字不一致之处。
- **关键点/保护点归纳**：按用户偏好的“本申请针对……设计……通过……实现……”风格成对输出。
- **附图请求**：先从最新文本确认模块顺序和任务范围，再生成黑白图。

## DOCX、PPTX 与公式处理

- DOCX/PPTX 作为输入时，先抽取正文、备注、表格、图题和公式附近文字，必要时转 Markdown 后扫描。
- 用户要求 Word 公式时，应尽量使用可编辑 Word 公式或 MathType 兼容对象，并进行显示检查。
- 用户允许 LaTeX 文本时，应保证 LaTeX 语法正确，并在相邻段落解释所有符号。
- 不得留下“公式待补充”、未解释变量或正文没有提及的公式符号。
- 修改已有 DOCX 时保留用户已有改动；用户要求高亮时，对新增或修改内容进行高亮标识。

## 语言风格

使用正式中文专利交底书语气。优先使用“本申请”“本提案”“该方法”“该模块”“在……阶段”等表述。避免“AI 赋能”“智能化提升”“革命性”“显著增强模型能力”等空泛或营销化措辞，除非后文给出具体技术机制和可验证技术效果。
